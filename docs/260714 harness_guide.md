# 에이전틱 엔지니어링 하네스 가이드

## 목적과 범위

이 문서는 개인 개발자가 자신의 작업 방식을 분해하고, AI 에이전트가 그 방식을 일관되게 재현·개선하도록 돕는 하네스의 설계 원칙을 정의한다. 기능 개발, 버그 수정, 문서화, 리뷰, 테스트, 배포 등 일반적인 개발 흐름에 적용할 수 있어야 한다.

처음부터 공용 프레임워크를 만드는 것은 목표가 아니다. 개인의 실제 작업에서 유용성과 안정성이 확인된 구성만 이후 공용화 후보로 승격한다. 이 문서는 고정된 설치 절차나 모든 작업을 강제하는 규정집이 아니라, 하네스를 구축·변경·검증할 때 사용하는 판단 기준이다.

## 핵심 모델: 네 계층

하네스는 다음 네 계층으로 구성한다.

```text
사용자 의도
    ↓
AGENTS.md ── 판단의 기본선·전역 협업 원칙
    ↓
Recipe Skill ── 작업 흐름을 선택·조합·단계화하는 기본 진입점
    ├── Skill ── 특정 전문 작업의 재사용 절차
    └── Memory ── 판단에 필요한 프로젝트 지식·사례·근거
```

### AGENTS.md: 프로젝트 헌장

`AGENTS.md`는 사람과 에이전트의 프로젝트 수준 협업 원칙을 제공한다. 의도 확인, 범위 통제, 기존 코드 존중, 작은 변경 선호, 검증, 가정과 위험의 기록처럼 모든 작업에 적용되는 판단의 기본선을 담는다.

세부 플레이북, 변하기 쉬운 프로젝트 사실, 특정 작업의 상세 순서는 넣지 않는다. 의미 있는 변경은 구체적 사용자 지시 또는 사용자가 수락한 명세·계획을 근거로 진행하며, 직접 사용자 지시가 최우선이다. 상세 반복 절차는 Skill·Recipe에, 프로젝트 지식은 Memory에 둔다.

### Recipe Skill: 판단 가능한 워크플로의 진입점

Recipe는 사용자의 요청이 들어왔을 때 작업 흐름의 기본 진입점이 된다. 단순한 Skill 묶음이나 고정 체크리스트가 아니라, 요청·현재 코드·프로젝트 지식에 따라 필요한 Skill과 Memory를 선택하고 경로를 조정하는 오케스트레이터다.

예를 들어 기능 개발 Recipe는 변경의 영향 범위, 새 도메인 규칙, 외부 의존성, 다중 진입점, 장기 확장 부담을 평가한다. 해당 신호가 충분할 때만 아키텍처 Memory와 경계 설계 Skill을 적용한다. 기존 경계 안의 작은 변경은 최소 설계·구현·관련 테스트·검증의 경로로 끝낼 수 있다.

따라서 Recipe는 LLM의 판단을 제한하는 대신 그 판단에 필요한 기준과 선택지를 제공한다. 각 Recipe는 최소한 다음 계약을 명시한다.

- 목적과 진입 조건
- 판단 및 분기 기준
- 조회할 Memory의 질의와 적용할 Skill
- 사용자 승인, 질문, 중단 지점
- 산출물과 완료·검증 조건
- 사용자 지시나 현재 저장소 구조에 따라 경로를 조정하는 이탈 규칙

### Skill: 단일 전문 작업의 재사용 절차

Skill은 테스트 설계, 코드 리뷰, 마이그레이션처럼 한 종류의 전문 작업에 집중한다. 입력, 절차, 산출물, 검증 방식, 안전한 실패·중단 조건을 가진다. 특정 저장소의 변하기 쉬운 사실은 Skill에 복제하지 않고 Memory를 통해 참조한다.

입력이 부족하거나 적용 조건이 맞지 않으면 Skill은 구현을 강행하지 않는다. 사용자에게 질문하거나, 상위 Recipe의 판단 단계로 되돌아간다.

### Memory: 지속 지식 라이브러리

Memory는 대화 상태나 에이전트를 통제하는 지시문이 아니다. 현재 저장소에서 무엇이 사실인지, 왜 그렇게 결정되었는지, 어떤 구현 사례가 검증되었는지를 제공하는 문서 시스템이다.

우선 다음 유형으로 구분한다.

- `decision`: 아키텍처·설계 결정, 대안, 선택 이유, 제약, 재검토 조건
- `implementation`: 모듈 구현 방식, 설정을 읽는 방법, 통합 절차, 주의점
- `example`: 검증된 실제 구현 위치와 재사용 가능한 조건

각 Memory에는 주제, 유형, 적용 조건, 출처, 갱신일, 소유자, 관련 Recipe·Skill을 기록한다. 사실·결정·예제를 구분하고, 코드 또는 최신 결정과 충돌하면 원문과 코드를 우선 확인한다.

## Memory Index와 지식 검색

에이전트가 파일명을 추측해 Memory를 직접 찾도록 하지 않는다. 얇은 메타 인덱스인 **Memory Index**가 모든 Memory의 제목, 유형, 태그, 적용 조건, 갱신 상태, 출처, 관련 Recipe·Skill을 카탈로그로 제공한다.

```text
Recipe / Skill
    ↓ 필요한 지식의 질의(작업·결정·기술·영역)
Memory Index
    ↓ 후보와 적합성·우선순위 제시
관련 Memory 원문
    ↓ 사실·결정·근거·예제 제공
LLM의 상황별 판단
```

Recipe와 Skill은 구체적인 Memory 파일을 무조건 나열하기보다, 필요한 지식의 질의를 선언한다. 예를 들어 인증 변경에는 `auth`, `security`, `configuration` 태그의 최신 `decision`·`implementation` Memory를 찾도록 지정할 수 있다. Index는 후보를 좁히는 수단일 뿐이며, 원문이 최종 근거다.

## 지시와 책임의 경계

지시의 우선순위는 다음과 같다.

1. 직접적인 사용자 지시
2. `AGENTS.md`
3. 선택된 Recipe와 Skill
4. Memory

하위 계층은 상위 원칙을 구체화할 수 있지만, 상위 원칙을 넓히거나 뒤집을 수 없다. Memory는 지시가 아닌 근거이며, Recipe는 작업 흐름을 조정하되 사용자의 구체적 의도를 대체하지 않는다.

## Memory의 수명주기와 사람의 승인

작업 중 생성된 요약·관찰·후속 메모는 기본적으로 임시 기록이다. 임시 기록은 영속 Memory처럼 자동 검색·참조 대상이 아니다.

LLM은 작업 종료 시 재사용 가능하고 검증 가능한 지식을 영속 Memory 후보로 제안할 수 있다. 제안에는 대상 유형, 근거, 적용 범위, 초안 위치, 기존 Memory와의 관계를 포함한다. 영속 Memory와 Memory Index의 생성·변경은 사람이 승인·수정·거절·보류한 뒤에만 반영한다. 이 책임은 사람에게 있으며, LLM은 제안자다.

## 하네스의 검증과 개선

하네스도 변경·검증 가능한 시스템으로 다룬다.

- Recipe·Skill·Memory 변경에는 목적과 영향 범위를 남긴다.
- Recipe는 진입 조건, 필요한 Skill·Memory의 발견 가능성, 승인·검증 단계의 존재를 점검한다.
- Memory는 Index 등록 여부, 출처·갱신일·소유자, 코드·결정과의 불일치를 검토한다.
- 작업 후에는 결과뿐 아니라 선택한 Recipe·Skill·Memory가 적절했는지를 짧게 회고한다.
- 회고에서 나온 지식은 LLM이 후보로 제안할 수 있으나, 사람 승인 전에는 영속화하지 않는다.

검증의 목적은 절차 준수 자체가 아니라, 올바른 지식과 적절한 작업 경로를 선택했는지 확인하는 것이다.

## 점진적 구축 원칙

하네스는 실제 사용에서 검증된 요소만 확장한다.

1. `AGENTS.md`로 전역 판단 원칙을 고정한다.
2. 자주 쓰는 한두 개의 Recipe를 기본 진입점으로 만든다.
3. Recipe에서 반복되는 전문 절차를 독립 Skill로 추출한다.
4. 검증된 결정과 구현 사례만 사람 승인 후 Memory와 Index로 승격한다.
5. 실제 작업 회고로 분기 기준, 검색 규칙, 검증 조건을 다듬는다.
6. 여러 프로젝트에서 안정적으로 재사용된 요소만 공용 프레임워크 후보로 검토한다.

이 순서는 개인의 작업 방식을 관찰해 하네스를 다듬는 것을 우선하며, 검증되지 않은 추상화와 성급한 공용화를 피한다.
